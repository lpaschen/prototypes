📐 Design decisions
Tier context

AI is given the user's current plan, usage, and full tier comparison at prompt time. It only surfaces the comparison when explicitly asked — not on every message.

In-app link strategy

Option 2 (Structured signal): The API returns a plan_cta field alongside the message text. The frontend renders the plan comparison card as a native UI component — not embedded in the AI text.

Benefit

The AI response stays readable in any context (API consumers, mobile, email). The card only appears in-product.

❓ Open questions
Q1
Should the full tier comparison show when the user asks about their plan, or only a delta ("here's what you'd gain")? A delta is less overwhelming but may miss context.
Q2
Should pricing appear in the AI response text, or only in the frontend-rendered card? Pricing can change — a card sourced from the billing API stays accurate.
📐 Design decisions
Feature gate card

When a user asks about a gated feature, the AI explains what it does and how to use it — then a feature gate card is rendered showing exactly which tiers unlock it.

Tone

The AI still explains the feature fully, rather than blocking with "you can't do that." Users deserve to understand what they're being asked to pay for.

Workaround

When a workaround exists, the AI surfaces it honestly — even if it's a friction point. This builds trust rather than pushing the upgrade hard.

❓ Open questions
Q1
Should the feature gate card always show all tiers, or just the user's current tier and the next tier up? Showing all tiers could cause noise.
Q2
If a user asks about multiple gated features in one conversation, should we show a gate card each time? Might feel pushy — consider showing only once per session.
📐 Design decisions
Proactive threshold

Warning is triggered when usage is ≥80% of the monthly cap. This fires once per session — not on every message — to avoid fatigue.

Context-aware

The AI detected the user was about to create a link (a usage-consuming action) and surfaced the warning before the creation step — giving them a chance to upgrade first.

Not blocking

The warning is informational, not a gate. The user can dismiss it and still create the link. This respects user autonomy.

❓ Open questions
Q1
Should the proactive warning fire at 80% or only when fewer than N links remain (e.g., <10 links)? A percentage may be misleading for users on large plans.
Q2
What's the right suppression logic? "Once per session" may be too frequent if the user is creating many links in a session.
📐 Design decisions
Hard limit

When the limit is already hit, the AI explains the error immediately and unambiguously — no guessing, no generic "something went wrong."

Two clear paths

The upgrade CTA offers both options: upgrade now, or wait for the reset. Users shouldn't feel forced — but both paths are surfaced clearly.

Accurate info

The AI correctly explained that deleting links doesn't restore quota — this is the kind of accurate entitlement knowledge the system prompt change enables.

❓ Open questions
Q1
Should the upgrade CTA include pricing ($29/mo) directly in the AI response? Or is that better left to the frontend card which can pull live pricing?
Q2
Should there be a way to request a one-time limit exception or overage purchase, rather than a full plan upgrade?
